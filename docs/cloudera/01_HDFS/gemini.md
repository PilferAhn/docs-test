# HDFS (Gemini 답변)

> 출처: Gemini 답변 원문

## 1. 기본 개념: "나눠서, 복사해서, 흩어놓는다"

일반적인 PC는 하나의 HDD/SSD에 파일을 저장합니다. 하지만 데이터가 수십 테라바이트(TB)를 넘어가면 물리적으로 한 대에 담을 수 없습니다. HDFS는 이를 해결하기 위해 다음과 같은 전략을 씁니다.

1. **블록(Block) 단위 분할:** 아주 큰 파일(예: 1TB)을 작은 조각(기본 128MB 또는 256MB)으로 잘게 쪼갭니다.
2. **분산 저장:** 쪼개진 조각들을 네트워크로 연결된 여러 대의 서버(DataNode)에 나누어 저장합니다.
3. **복제(Replication):** 컴퓨터 한 대가 고장 나도 데이터를 보존하기 위해 각 조각을 기본적으로 **3군데의 서로 다른 서버에 복사**해 둡니다.

---

## 2. 파일 및 폴더 구조: HDD/SSD와 같은 트리인가?

**네, 사용자가 보기에는 동일한 트리 구조입니다.**

- **논리적 구조 (사용자 관점):** 사용자는 HDFS 명령어(예: `hdfs dfs -ls /user/data`)를 통해 `/user/myfolder/file.txt`와 같은 경로로 파일에 접근합니다. 우리가 흔히 아는 윈도우 폴더 구조와 똑같습니다.
- **물리적 구조 (시스템 관점):** 하지만 실제 하드디스크 안에는 저 폴더가 존재하지 않습니다. 대신 수많은 '블록 파일'들이 번호표를 달고 여러 서버의 로컬 디스크에 흩어져 있습니다.

---

## 3. HDFS의 핵심 구성 요소 (Master-Slave 구조)

### ① 네임노드 (NameNode) - "도서관 사서"

- **역할:** 파일 시스템의 **메타데이터(설계도)**를 관리합니다.
- **저장 정보:** 어떤 파일이 어떤 폴더에 있는지, 그 파일은 어떤 블록들로 구성되어 있는지, 각 블록은 몇 번 서버(DataNode)에 저장되어 있는지 등을 기록합니다.
- **특징:** 메모리(RAM) 상에서 이 정보를 관리하기 때문에 매우 빠르지만, 네임노드가 죽으면 전체 데이터에 접근할 수 없게 됩니다.

### ② 데이터노드 (DataNode) - "책꽂이 창고"

- **역할:** **실제 데이터(블록)**를 저장하는 서버입니다.
- **특징:** 클라이언트가 파일을 읽거나 쓰고 싶을 때, 네임노드에게 위치를 물어본 뒤 실제 데이터는 이 데이터노드와 직접 주고받습니다.

---

## 4. 파일 저장 프로세스 (쓰기 시나리오)

사용자가 `movie.mp4` 파일을 HDFS에 저장할 때의 과정입니다.

1. **요청:** 클라이언트가 네임노드에게 "파일 좀 저장할게"라고 말합니다.
2. **계획 수립:** 네임노드는 파일 크기를 확인하고 "A, B, C 블록으로 쪼개서 1번, 5번, 9번 서버에 각각 저장해"라고 응답합니다.
3. **전송:** 클라이언트는 첫 번째 데이터노드에 데이터를 보냅니다.
4. **파이프라인 복제:** 데이터를 받은 첫 번째 데이터노드는 두 번째로, 두 번째는 세 번째로 데이터를 복사합니다(복제본 생성).
5. **완료 보고:** 모든 복제가 끝나면 네임노드에게 "저장 완료!"라고 보고합니다.

## 5. HDFS의 특징과 한계점

### 장점

| 장점 | 설명 |
|:---|:---|
| 장애 복구 (Fault Tolerance) | 서버 한두 대가 고장 나도 복제본이 있으므로 데이터 유실이 없습니다 |
| 저사양 하드웨어 활용 | 비싼 고성능 서버가 아니라 일반 사양의 서버(Commodity Hardware) 여러 대를 묶어 대용량 저장소를 만들 수 있습니다 |
| 무한 확장 (Scale-out) | 서버만 추가하면 저장 용량을 무한대로 늘릴 수 있습니다 |

### 단점 및 주의사항

| 단점 | 설명 |
|:---|:---|
| 수정 불가 (Append Only) | 한 번 저장된 파일은 수정(Update)이 어렵습니다. 파일 끝에 데이터를 덧붙이는 것(Append)만 가능합니다 |
| 작은 파일 문제 (Small File Problem) | 네임노드는 모든 파일의 정보를 메모리에 저장합니다. 아주 작은 파일이 수억 개가 되면 네임노드의 메모리가 버티지 못합니다 |
| 빠른 응답 속도 부족 | 실시간 DB처럼 아주 빠르게 읽고 쓰는 용도보다는 대용량 데이터를 한 번에 읽어서 분석하는 배치(Batch) 용도에 적합합니다 |

## Gemini 관점 요약

> HDFS는 Gemini 초기 답변에서 독립 서비스로 비교되지 않았으나,
> 추가 질문을 통해 위와 같이 상세 설명을 받았습니다.
> CDW(데이터 웨어하우징) 맥락에서는 다음과 같이 언급되었습니다.

- Cloudera CDW는 Apache Hive(LLAP)와 Impala 엔진을 사용하며, 이 엔진들이 HDFS 위의 데이터를 읽는 구조입니다.
- Databricks SQL은 클라우드 오브젝트 스토리지(S3/ADLS/GCS) 위의 Delta Lake 테이블을 대상으로 동작합니다.
- Cloudera의 핵심 엔진(Hive, Impala, Spark, NiFi) → HDFS를 저장소로 사용
- Databricks의 핵심 엔진(Apache Spark, Photon) → 클라우드 오브젝트 스토리지 사용
