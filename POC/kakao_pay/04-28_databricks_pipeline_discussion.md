# Databricks 데이터 파이프라인 구축 및 운영 기술 심층 논의

---

## 초기 데이터 적재를 위한 환경 설정 및 아키텍처

고객사 AWS 환경의 S3 버킷에 있는 데이터에 안전하게 접근하고 Databricks로 가져오기 위해 필요한 초기 환경 설정 및 아키텍처 구성 방안을 논의했습니다. Databricks 워크스페이스는 Databricks 사의 VPC에 위치한 웹 서버를 통해 접속하며, 사용자의 데이터는 고객사 VPC 내에 안전하게 보관됩니다.

워크스페이스 접근을 제어하기 위해 IP 액세스 리스트를 설정해야 하며, 여기에는 카카오페이 IP, 데이터베이스의 VPN IP, 그리고 우리 회사 IP가 등록될 예정입니다. 데이터 접근 권한 관리는 AWS의 IAM(Identity and Access Management) Role과 STS(Security Token Service)를 통해 이루어집니다. 사용자가 워크스페이스에서 클러스터 생성을 요청하면, IAM Role에 부여된 권한을 통해 AWS 리소스(S3 버킷 등)에 접근하게 됩니다.

데이터 접근 시나리오는 다양합니다. 클러스터 생성에 사용되는 IAM Role이 직접 S3 버킷에 접근할 수도 있고, 보안 강화를 위해 다른 특정 Role의 권한을 빌려서(Assume Role) 접근할 수도 있습니다. 후자의 경우, 권한을 빌려 쓸 Role을 Databricks의 스토리지 크리덴셜(Storage Credential)과 외부 위치(External Location)에 명시적으로 등록해야 합니다. 이 설정 작업은 보통 SA(Solution Architect)가 수행하지만, 필요시 엔지니어가 직접 처리할 수 있으며, 이는 프로젝트 신뢰도를 높이는 데 기여합니다.

## COPY INTO를 활용한 데이터 적재 실행 방안

초기 및 증분 데이터 적재를 효율적으로 수행하기 위한 COPY INTO 명령어의 활용법을 중점적으로 다루었습니다. 이 명령어는 테이블명, 파일 포맷(주로 Parquet), 그리고 데이터가 위치한 S3 경로를 지정하여 대용량 데이터를 테이블로 적재하는 데 사용됩니다. 초기 데이터 양이 많을 때 특히 유용합니다.

COPY INTO의 핵심 기능 중 하나는 자동 체크포인트 관리입니다. 이는 Autoloader와 유사하게 작동하며, 한번 실행된 파일은 기록해두었다가 다음 실행 시 새로운 파일만 자동으로 감지하여 중복 처리 없이 증분 적재를 수행합니다. 이 체크포인트 정보는 대상 테이블 하위에 자동으로 관리되므로 별도의 경로 지정이 필요 없습니다. 이러한 안정성 덕분에 여러 번 실행해도 항상 동일한 결과를 보장합니다.

COPY INTO를 통해 생성된 테이블은 일반적으로 Raw 데이터가 담긴 브론즈(Bronze) 테이블이 됩니다. 이 초기 적재 작업은 데이터 양이 많아 시간이 소요될 뿐, 기술적으로 어려운 작업은 아니라고 평가되었습니다. 데이터 적재 파이프라인의 트리거는 COPY INTO 명령어 자체라기보다는, 해당 명령어를 실행시키는 스케줄링 잡(Job) 또는 외부 이벤트가 됩니다. 생성된 파이프라인은 스케줄링을 통해 주기적으로 실행하거나, 실시간 데이터 유입을 감지하는 연속(Continuous) 모드로 운영할 수 있습니다.

## SDP를 활용한 데이터 변환 및 품질 관리

적재된 Raw 데이터(브론즈)를 비즈니스 로직에 맞게 정제된 데이터(실버, 골드)로 변환하고 데이터 품질(DQ, Data Quality)을 보장하기 위해 SDP(Declarative Pipelines, 구 DLT)를 활용하는 방안을 설명했습니다. SDP는 Python과 SQL을 모두 지원하며, UI를 통해 초기 코드 생성이 가능하여 개발 편의성을 높입니다.

데이터 품질 관리를 위해 Expectation이라는 규칙을 사용합니다. SQL 또는 Python 코드로 데이터가 만족해야 할 조건을 명시하고, 조건에 맞지 않는 데이터를 누락(drop)시키거나 파이프라인을 실패(fail)시키는 등의 조치를 정의할 수 있습니다. 이러한 규칙은 딕셔너리(Key-Value) 형태로도 정의할 수 있으며, 규칙의 이름(Key)은 사용자가 자유롭게 지정 가능합니다. 특히 `dp.expect_all`과 같은 일부 함수는 Databricks 전용 기능으로, 오픈소스 스파크에서는 제공되지 않습니다.

SDP로 정의된 파이프라인은 데이터베이스에서 실시간 스트리밍 처리를 의미합니다. 트리거 종류는 한 번만 실행되는 배치 모드와, 소스 데이터의 변경을 지속적으로 감지하여 처리하는 연속(Continuous) 모드가 있습니다. COPY INTO를 실행하는 작업이 새로운 데이터를 유입시키는 트리거가 되고, 이어서 SDP 파이프라인이 후속 데이터 변환 및 검증 작업을 수행하는 구조로 설계할 수 있습니다.

## 데이터 처리 성능 최적화 및 기술 심화

대용량 데이터 처리 시 성능을 최적화하기 위한 고급 기술들을 심도 있게 다루었습니다. 클러스터 스펙 산정 시, 처리 속도가 중요하다면 데이터 양의 약 3배에 해당하는 총 워커 메모리를 확보하는 것이 좋습니다. 예를 들어 100GB 데이터 처리 시, 총 300GB 메모리(예: 64GB 워커 5~6대)를 구성하면 한 번의 사이클로 처리가 가능합니다. 또한, 작은 스펙의 서버 여러 대보다 큰 스펙의 서버를 적게 두는 것이 분산 처리 효율에 더 유리합니다.

운영 중 스키마가 변경되는 '스키마 에볼루션'은 몇 가지 제약이 따릅니다. 컬럼의 데이터 타입이 더 큰 범위로 변경되는 '타입 와이드닝'(예: INT → BIGINT, DATE → TIMESTAMP)만 지원되며, 더 작은 범위로의 변경은 데이터 유실 위험 때문에 허용되지 않습니다.

데이터 저장 시 사용되는 압축 알고리즘은 성능과 비용에 큰 영향을 미칩니다. 데이터는 일반적으로 원본의 약 1/5 크기로 압축되어 저장되며, Databricks는 기본적으로 Zstandard, 스파크는 Snappy를 사용합니다. Parquet 파일 포맷은 파일을 분할하여 병렬로 읽을 수 있어 CPU 코어 수가 많을수록 초기 적재 속도가 향상됩니다. 반면, Gzip은 분할 읽기는 불가능하지만 여러 압축 파일을 이어 붙일 수 있고, 압축 레벨을 높여 압축률을 극대화할 수 있으나 처리 속도는 느려집니다. 이러한 기술적 특성을 이해하고 워크로드에 맞는 최적의 조합을 선택하는 것이 중요합니다.
