# 카카오페이 데이터 플랫폼 마이그레이션 PoC — 1차 정리

> **목적:** Cloudera 온프레미스 데이터 플랫폼 → Databricks Lakehouse 전환 가능성 종합 검증
> **기간:** 총 6주 (기술지원 인력 3주 투입)

---

## 1. AS-IS 현황 (현재 시스템)

### 플랫폼 구성

```
Cloudera (온프레미스 Hadoop 기반)
├── Hive Metastore       ← 테이블 스키마/경로 정보 저장소 (MySQL 기반)
├── Hive Table           ← HDFS 위에 SQL로 접근하는 테이블
├── Kudu Table           ← Cloudera 전용 컬럼형 스토리지 (실시간 UPDATE 지원)
└── Impala               ← 빠른 SQL 쿼리 엔진

Iceberg / Tiara
├── CEPH                 ← RedHat의 S3 호환 오브젝트 스토리지 (HDFS 대신 사용)
├── Iceberg 포맷         ← 주력 테이블 포맷 (Trino 호환성 때문으로 추정)
└── Hive Metastore       ← 스키마 정보 공유

쿼리 엔진
├── Impala               ← Hive 기반 고속 SQL
├── Trino / Starburst    ← Trino 상용 버전, 현재 사용 중으로 추정
└── StarRocks            ← 초저지연 OLAP (100ms 미만 목표)

수집 / 파이프라인
├── Kafka                ← 실시간 메시지 스트리밍
├── Debezium             ← CDC 오픈소스 도구 (MySQL binlog → Kafka)
├── Minecraft            ← 내부 CDC 파이프라인 도구 (명칭 확인 필요)
└── Airflow              ← 배치 파이프라인 스케줄링

기타
├── MSTR (MicroStrategy) ← BI 시각화 도구
├── DataHub              ← 데이터 카탈로그
├── Keycloak             ← 사내 SSO 인증 서버
├── QueryPie             ← 데이터 접근 통제 및 감사 도구
└── Apache Ranger        ← Cloudera 기반 보안/권한 관리
```

### 데이터 규모

```
티아라 로그        230GB/day, 1개월 7TB, 1년 84TB
ETL 배치          36TB (45개 테이블, 최근 1개월)
결제 CDC          초당 600건, Kafka lag 12시간 = 2,600만 건
티아라 클릭스트림  초당 4.5만 건
ML 대상           2,000만 유저 × 200 feature
스키마 변경 테스트 7TB 테이블 1개
─────────────────────────────────────────
명시적 합계       최소 120TB+  (Kudu, ML 등 미포함)
```

## 2. TO-BE 전환 목표 (Databricks)

```
AS-IS (Cloudera 온프레미스)       TO-BE (Databricks + AWS)
──────────────────────────────────────────────────────────
HDFS / CEPH               →   S3
Hive Metastore            →   Unity Catalog
Kudu                      →   Delta Lake
Impala / Trino(Starburst) →   Databricks SQL / Photon
Hive ETL                  →   Databricks Workflow / DLT
Airflow                   →   Databricks Workflow + Airflow Operator
Apache Ranger             →   Unity Catalog 권한 정책
MSTR BI                   →   Databricks SQL Warehouse 연동
온프렘 ML 파이프라인       →   Databricks MLflow + Feature Store
Keycloak SSO              →   Keycloak ↔ AWS SSO ↔ Databricks SSO
```

## 3. 핵심 기술 요건 및 현실성 검토

### 요건 1 — 초고속 쿼리 응답 (난이도: 매우 높음)

```
요구사항   StarRocks 기준 100ms 미만 응답
현실       일반 DB 통상 2~3초, 100ms는 사람이 거의 인지 못하는 수준
판단       Databricks SQL Warehouse로 달성 어려울 수 있음
           → 별도 아키텍처 검토 필요 (Serverless SQL + Photon)
           → 목표 완화 협의 필요
```

### 요건 2 — 대규모 탈퇴자 삭제 / GDPR (난이도: 높음, PoC 핵심)

```
1개월 삭제   225GB, 탈퇴자 1만 건, 24시간 이내
1년 삭제     84TB,  탈퇴자 12만 건, 48시간 이내

주의사항
├── 12만 건은 탈퇴 유저 수 → 실제 삭제 데이터는 훨씬 방대
├── Delta Lake는 PK/FK/인덱스 없음 → 전체 테이블 스캔 필요
├── 삭제 중 실시간 수집과 충돌 검증 필요
└── Delta 삭제 성능이 Databricks 도입 판단의 핵심 변수
```

### 요건 3 — 실시간 클릭스트림 처리 (난이도: 중간)

```
규모       초당 4.5만 건, 230GB/day
파이프라인  Kafka → Spark Structured Streaming → Delta Lake
검증       Exactly-Once 보장, Small File 문제, Compaction 주기
```

### 요건 4 — CDC 장애 복구 (난이도: 중간~높음)

```
시나리오   MySQL binlog → Debezium → Kafka → DLT
규모       결제 초당 600건, lag 12시간 = 2,600만 건 누적
검증       Exactly-Once, Schema Evolution, Kafka lag 복구
확인 필요  'Minecraft' = 내부 CDC 도구인지 고객 확인 필요
```

### 요건 5 — ML 모델 이관 (난이도: 낮음, PoC 범위 한정)

```
PoC 범위   신규 모델 개발 아님
           기존 모델 파일 (Pickle, H5 등) → MLflow 등록 → 서빙/추론 검증
필요 사항  고객 측 모델 파일 S3 업로드 필요
협의 필요  Feature Store, 모델 모니터링 → 실구현 vs 문서 대체 결정
```

## 5. 비즈니스 / 규제 환경

### 온프레미스 → 클라우드 전환 배경

```
서버 교체 주기  통상 5년, 수명 한계 7년
서버 비용 급등  최근 5배 상승 → 클라우드 유지비와 비슷해짐
초기 도입 비용  클라우드는 없음 → 전환 가속 요인
```

### CSP 안전성 평가 (금융권 규제)

```
주관       금융감독원 (금감원)
근거       금융분야 클라우드 이용 가이드라인 (금융위원회 고시)
프로세스   금융회사 자체 평가 → 금감원 신고 → 승인 → 사용 가능

Databricks 현황
├── CSP 안전성 평가   통과 완료 → 기본 기능 사용 가능
└── Serverless 기능   평가 진행 중 → 통과 전까지 금융사 사용 불가
                      → 아키텍처 설계에 영향
```

### 국가핵심기술 제약

```
해당 케이스  삼성전자 반도체 등 국가 전략 산업
제약         생산 데이터 외산 클라우드(AWS 등) 저장 불가
대안         SCP(삼성SDS), NCP(네이버클라우드) 등 국내 클라우드
한계         Databricks가 국내 클라우드 미지원 → 도입 불가 구조
카카오페이   금융업 → 위 제약 해당 없으나 CSP 평가 필수
```

## 6. PoC 우선순위

### 1순위 — 데이터 플랫폼 핵심
```
□ HDFS / Kudu / Hive / Iceberg → Delta 전환
□ Kafka → Delta Streaming (티아라 클릭스트림)
□ CDC 처리 (MySQL → Debezium → Kafka → DLT)
□ 36TB 배치 ETL 파이프라인 전환
□ SQL 성능 비교 (Impala/Trino/StarRocks vs Databricks SQL/Photon)
□ 탈퇴자 삭제 성능 검증 (GDPR)
```

### 2순위 — 운영 / 보안
```
□ Unity Catalog 권한/마스킹 (Apache Ranger 대체)
□ QueryPie 연동
□ Audit Log
□ SSO (Keycloak ↔ AWS SSO ↔ Databricks)
□ Workflow 승인 연동
```

### 3순위 — BI / 분석
```
□ MSTR ↔ Databricks SQL Warehouse
□ 큐브 쿼리 성능, 동시 사용자 테스트
```

### 4순위 — 비용 최적화
```
□ Serverless vs Classic (CSP 평가 통과 여부에 달림)
□ Spot Instance, Cluster Policy
□ OPTIMIZE / VACUUM
```

### 5순위 — ML / MLOps
```
□ 기존 모델 파일 MLflow 등록 및 추론 검증
□ Feature Store, Batch Inference (범위 협의 필요)
```
