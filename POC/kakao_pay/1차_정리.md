# 카카오페이 데이터 플랫폼 마이그레이션 PoC — 1차 정리

> **목적:** Cloudera 온프레미스 데이터 플랫폼 → Databricks Lakehouse 전환 가능성 종합 검증
> **기간:** 총 6주 (기술지원 인력 3주 투입)
> **문서 성격:** overview + 1차 미팅 교차 분석 + 구조적 패턴으로 예측한 PoC 접근법

---

## 1. 두 문서를 교차하면 명확해지는 것들

---

### 1-1. 실제 스토리지는 HDFS가 아니라 CEPH

```
overview 표현
└── "HDFS External Table" → HDFS에 저장하는 것처럼 기술

실제 (미팅)
└── CEPH 사용 중 (RedHat의 S3 호환 오브젝트 스토리지)

의미
└── 이관 경로가 "HDFS → S3"가 아니라 "CEPH → S3"
    CEPH가 이미 S3 호환이므로 DistCp보다 S3 호환 API 직접 활용 가능성
```

---

### 1-2. Trino는 비교 대상이 아니라 이미 사용 중

```
overview 표현
└── Trino를 비교 항목으로만 기술, AS-IS에 Impala만 명시

실제 (미팅)
└── Starburst (Trino 상용 버전) 실제 운영 중
    Iceberg 포맷을 주력으로 쓰는 이유 = Trino 호환성

의미
└── 쿼리 엔진이 Impala + Starburst 병행 구조일 가능성
    Iceberg → Delta 전환 시 Starburst 호환성 깨질 수 있음
    UniForm 검증이 더 중요해짐
```

---

### 1-3. 100ms 요구사항 — 달성 가능성 낮음

```
overview 표현
└── StarRocks와의 단순 비교 항목으로 기술

실제 (미팅)
└── 고객이 StarRocks에서 이미 경험한 속도를 기대하는 hard requirement
    일반 DB 통상 2~3초, 100ms는 사람이 거의 인지 못하는 수준

의미
└── Databricks SQL Warehouse로 달성 어려움
    PoC에서 "불가" 결론이 나올 수 있는 항목
    사전에 기대치 조율 필수
```

---

### 1-4. Serverless 기능 — 비용 최적화 시나리오 전체에 영향

```
overview 표현
└── 비용 시나리오에서 "Classic vs Serverless" 당연히 쓸 수 있는 것처럼 기술

실제 (미팅)
└── Serverless는 CSP 안전성 평가 진행 중
    평가 통과 전까지 금융사 사용 불가

의미
└── 비용 최적화 핵심 옵션이 현재 막혀 있음
    PoC에서 Serverless 검증 자체가 불가할 수 있음
    아키텍처 설계 시 Serverless 의존도 낮춰야 함
```

---

### 1-5. ML/MLOps — 문서 범위와 실제 PoC 범위의 괴리

```
overview 표현
└── BM Index, LAL, Feature Store, Vector Search, Real-time Serving 등 방대한 항목

실제 PoC 범위 (미팅)
└── 기존 모델 파일(Pickle, H5) → MLflow 등록 → 추론 가능한지 수준
    Feature Store, 모델 모니터링은 실구현 vs 문서 대체 협의 중

의미
└── overview ML 시나리오는 희망 목록에 가까움
    고객 기대치와 실제 범위 간 갭 조율 필수
```

---

### 1-6. 탈퇴자 삭제 — 단순 삭제 성능이 아닌 복합 검증

```
실제 조합
└── 84TB 삭제 + 동시에 초당 4.5만 건 스트리밍 수집 진행 중
    Delta Lake는 PK/FK/인덱스 없음 → 전체 테이블 풀스캔
    12만 건 = 탈퇴 유저 수 → 실제 삭제 데이터는 훨씬 방대

의미
└── 이 조합이 실패하면 Databricks 도입 자체가 불가
    PoC 설계에서 가장 먼저, 가장 신중하게 다뤄야 할 항목
```

---

### 1-7. 문서가 여러 팀의 요구사항을 합친 것

```
미팅에서 파악
└── 문서 작성 형식이 3가지로 나뉨
    → 플랫폼 관리팀, 현업 분석팀, ML팀 등 여러 팀의 wish list 통합

의미
└── 범위가 넓은 이유 = 팀별 요구사항이 우선순위 없이 합쳐진 것
    팀별로 핵심 요구사항이 다름
    → PoC 킥오프 전 팀별 우선순위 재합의 필요
```

## 2. 비유에서 보이는 구조적 패턴 → 다른 항목 예측

> 회의에서 등장한 비유들을 보면 기술 구조의 반복 패턴이 보임
> 이 패턴을 다른 항목에 적용하면 PoC 접근법이 예측됨

---

### 패턴 1: "중간에 껴있는 변환기" 구조

```
비유의 근거
└── Debezium = MySQL 내부 언어(binlog) → Kafka 공통 언어로 변환하는 통역사
    Kafka Broker = 물류 창고 (Producer/Consumer 사이의 중간 보관소)
    Hive Metastore = 주소록 (데이터 위치만 알려줌, 실행은 안 함)

이 패턴이 반복되는 다른 항목
├── Kudu → Spark Kudu Connector → Delta
│         ↑ Debezium과 동일한 역할, 변환기
│         Kudu 자체 스토리지를 Spark이 읽어서 변환
│
├── MSTR → JDBC/ODBC 드라이버 → Databricks SQL Warehouse
│          ↑ MSTR 자체는 안 건드림, 연결 대상만 교체
│          기존 Impala JDBC → Databricks JDBC로 변경
│
└── Keycloak → AWS SSO → Databricks SSO
              ↑ Keycloak 자체는 안 건드림, 연결 체인만 추가

예측되는 PoC 접근법
└── "기존 시스템을 교체"하는 게 아니라
    "기존 시스템과 Databricks 사이에 변환기/연결고리 추가"하는 방식
    → 기존 시스템은 최대한 건드리지 않음
```

---

### 패턴 2: Kafka는 온프렘에 그대로, Consumer만 교체

```
비유의 근거
└── Kafka = 물류 허브 (판매자/배달기사 서로 몰라도 됨)
    → Producer(티아라, MySQL)와 Consumer(Databricks)는 독립적

현재 구조
└── 티아라 → Kafka (온프렘) → 기존 Consumer (Spark/Flink)
    MySQL  → Debezium → Kafka (온프렘) → 기존 Consumer

PoC에서 예측되는 접근법
└── Kafka 온프렘 그대로 유지
    Consumer 쪽만 Databricks Structured Streaming / DLT로 교체
    VPN이 필요한 이유 = Kafka(온프렘) → Databricks(AWS) 연결 때문

의미
└── Kafka 자체를 AWS로 옮기는 작업은 PoC 범위 밖
    Kafka 브로커 주소만 알면 Databricks Consumer 붙이는 건 상대적으로 단순
    → CDC, 티아라 스트리밍 시나리오 모두 이 방식 적용
```

---

### 패턴 3: "자체 스토리지 있는 시스템"은 이관이 복잡

```
비유의 근거
└── Kudu = MySQL과 비슷 (자체 스토리지, 실시간 CRUD)
    CEPH/HDFS = 파일 시스템 (그냥 복사 가능)

자체 스토리지 있는 시스템들
├── Kudu    → 파일 직접 복사 불가, Spark Kudu Connector 경유 필수
└── MySQL   → 파일 직접 복사 불가, Debezium/CDC 경유 필수

파일 기반 시스템들
├── CEPH (Iceberg) → S3 호환이므로 직접 복사 가능
└── Hive Table     → HDFS 파일이므로 DistCp로 직접 복사 가능

예측되는 PoC 접근법
├── CEPH/Hive → DistCp 또는 S3 API로 직접 이관 (빠름)
└── Kudu → Spark Connector로 읽어서 Delta로 변환 (느림, 부하 있음)
    → Kudu 이관이 병목이 될 가능성 높음
```

---

### 패턴 4: Hive Metastore = 주소록 → Unity Catalog도 주소록인데 역할이 더 많음

```
비유의 근거
└── Hive Metastore = 주소록 (스키마/위치 저장, 실행 안 함)
    Impala/Starburst가 주소록을 보고 실제 쿼리 실행

현재 구조
└── Starburst → Hive Metastore 조회 → CEPH에서 Iceberg 파일 읽기

Unity Catalog로 전환 시
└── Starburst → Unity Catalog 조회 → S3에서 Delta 파일 읽기
    ↑ 이게 가능한지가 시나리오 K의 핵심 검증 항목
    "Trino에서 Databricks UC 질의" = Starburst가 UC를 주소록으로 인식할 수 있는가

예측되는 PoC 접근법
└── 완전 전환 전 하이브리드 기간 필요
    Starburst가 UC를 External Metastore로 바라보는 구조로 브릿지
    UniForm = Delta이면서 동시에 Iceberg로도 읽힘
    → Starburst는 Iceberg로 읽고, Databricks는 Delta로 읽는 공존 가능
```

---

### 패턴 5: Delta = Parquet + 장부 → 삭제/수정이 "파일 재작성"

```
비유의 근거
└── Delta = Parquet 파일 + 트랜잭션 로그 (장부)
    Time Travel = 장부 보고 과거 상태 복원
    Deletion Vectors = 파일 안 지우고 "이 행 삭제됨" 표시만

탈퇴자 삭제 시 일어나는 일
└── 84TB 전체를 스캔해서 해당 유저 행 찾기
    → 해당 파일을 새로 재작성 (삭제 행 제외)
    → 기존 파일을 구버전으로 표시
    → VACUUM 실행 전까지 구버전 파일이 디스크에 남음

예측되는 이슈
├── 재작성 대상 파일 수가 많을수록 시간 오래 걸림
│   → Liquid Clustering으로 탈퇴자 PK 기준 파일 집중 가능하면 빨라짐
├── 삭제 중 동일 파일에 스트리밍 write가 들어오면 충돌
│   → 행 수준 동시성(Row-level Concurrency) 검증이 핵심
└── VACUUM 전 구버전 파일이 남아있어 스토리지 일시적으로 2배 증가
    → 비용 계획에 반영 필요
```

---

### 패턴 6: CDC Kafka lag = 창고에 쌓인 미처리 택배

```
비유의 근거
└── Kafka lag = 배달기사(Consumer)가 처리 못해서 창고에 쌓인 택배
    장애 복구 = 배달기사 재투입 후 쌓인 것부터 순서대로 처리

12시간 lag = 2,600만 건 시나리오
└── 장애 발생 → Consumer 중단 → Kafka에 2,600만 건 쌓임
    Consumer 복구 → 2,600만 건을 한꺼번에 처리

예측되는 이슈
├── 복구 시 순간적으로 엄청난 처리 부하 → 클러스터 Auto-scaling 필요
├── 2,600만 건 처리 중 순서 보장 필요 (결제 데이터 특성상 순서 중요)
├── 처리 완료까지 실시간 테이블과 배치 테이블 간 데이터 불일치 구간 발생
└── 복구 완료 기준을 어떻게 정의할지 명확화 필요
    (lag = 0이 되는 시점 vs 데이터 정합성 검증 완료 시점)
```

## 3. AS-IS 실제 스택 (교차 분석 반영)

```
저장소
├── CEPH           ← HDFS 역할 대체 (S3 호환 오브젝트 스토리지)
└── Kudu           ← 실시간 CRUD 전용 (자체 스토리지)

테이블 포맷
└── Iceberg        ← 주력 포맷 (Trino/Starburst 호환성 때문)

쿼리 엔진
├── Impala         ← Hive 기반 SQL
├── Starburst      ← Trino 상용 버전 (실제 사용 중)
└── StarRocks      ← 초저지연 OLAP, 100ms 기준점

수집 파이프라인
├── Kafka          ← 메시지 스트리밍 허브 (온프렘 유지 예상)
├── Debezium       ← MySQL binlog → Kafka CDC 변환기
└── Minecraft      ← 내부 CDC 파이프라인 도구 (명칭 확인 필요)

메타데이터
└── Hive Metastore ← 주소록 역할 (MySQL 기반), 실제 SQL 실행 안 함

워크플로우
└── Airflow        ← 배치 스케줄링

보안/권한
├── Apache Ranger  ← Cloudera 기반 권한 관리
├── Keycloak       ← 사내 SSO
└── QueryPie       ← 데이터 접근 통제/감사

BI
└── MSTR           ← JDBC/ODBC로 Impala/Starburst에 연결
```

| 리스크 | 내용 | 판단 |
|:---|:---|:---|

| VPN/네트워크 | Kafka 온프렘 연결, CDC 실시간 테스트 필수 | 독립 대역폭 확보 필요 |

## 7. PoC 우선순위

### 1순위 — 도입 판단 핵심
```
□ 탈퇴자 삭제 + 동시 스트리밍 충돌 검증 (행 수준 동시성)
□ Kafka Consumer만 교체하는 방식으로 티아라 스트리밍 적재
```
